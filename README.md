# nx-typescript-7

Benchmark TypeScript 7 (native) against TypeScript 6 in one Nx workspace.

The `@nx/js/typescript` plugin is registered twice over all packages, so every project gets both compiler variants as separate targets:

| Target           | Compiler                    | Command                                            |
| ---------------- | --------------------------- | -------------------------------------------------- |
| `build`          | TypeScript 7 (native `tsc`) | `tsc --build tsconfig.lib.json`                    |
| `build-tsc6`     | TypeScript 6 (`tsc6`)       | `tsc6 --build tsconfig.lib.json`                   |

## Setup

TypeScript 7 ships a `tsc` binary but no JavaScript API, so tools (`@nx/js`, `typescript-eslint`, `vite`) still need TypeScript 6:

```json
{
  "devDependencies": {
    "@typescript/native": "npm:typescript@^7.0.2",
    "typescript": "npm:@typescript/typescript6@^6.0.2"
  }
}
```

- `typescript` alias stays on `@typescript/typescript6` so `require('typescript')` returns a compiler API. It provides the `tsc6` binary.
- `@typescript/native` exists only to supply the TS7 `tsc` binary.

This repo uses pnpm (`pnpm-workspace.yaml`, inter-package deps use `workspace:*`). pnpm doesn't hoist transitive binaries, so the npm-only hazard of `@typescript/old` stealing `.bin/tsc` doesn't apply -- but verify after swapping package managers anyway:

```shell
./node_modules/.bin/tsc --version   # Version 7.x
./node_modules/.bin/tsc6 --version  # Version 6.x
```

## Workspace

10 packages (`packages/lib001`..`lib010`) in a single dependency chain, generated by `tools/generate-packages.mjs`:

```shell
node tools/generate-packages.mjs --count 10 --per-layer 1 --files 120 --kinds 48
```

- Each package: 120 feature modules, each built around a 48-member discriminated union with mapped types and generic ops (~69k lines / ~1.9 MB of source per package). Checker-heavy on purpose -- see design notes.
- `--per-layer 1` makes the graph a chain (`lib010 -> lib009 -> ... -> lib001`); TS project references mirror it, `nx sync` clean.
- Knobs: `--count` (packages), `--per-layer` (graph width; >1 gives each package 2 deps from the previous layer), `--files`, `--kinds`.

### Design notes: what shape shows the TS7 speedup

Tuned empirically on an M-series (12 cores, 8 performance); per-package `tsc --build` ratios:

| Shape | tsc6 | tsc (TS7) | Ratio |
|:---|---:|---:|---:|
| 5 files, 10-kind unions (tiny) | 0.38s | 0.09s | 4.2x |
| 300 files, 12 kinds (parse/emit-heavy) | 6.6s | 0.95s | 6.9x |
| 900 files, 16 kinds | 24.7s | 3.5s | 7.1x (plateau) |
| 120 files, 48 kinds (checker-heavy) | 13.6s | 1.31s | **10.4x** |

1. **Checker-heavy code is where TS7 shines.** tsgo parallelizes type checking across files; wide discriminated unions + mapped types push the ratio to ~10x. Parse/emit-heavy code plateaus around 7x.
2. **The task graph must be deep, not wide.** With a wide graph, `nx run-many` (default `--parallel=3`) runs 3 single-threaded `tsc6` processes concurrently -- task-level parallelism gives TS6 the multi-core scaling it lacks internally, compressing the wall-clock ratio to ~3-4x. A chain forces one task at a time, so per-project compiler speed is all that matters (the monorepo critical-path story).
3. Tiny projects measure process startup + orchestration, not the compiler (a 100-package repo of ~300-line packages showed only 3x).

Both compiler variants share `tsconfig.lib.json` and emit to the same `dist/`, so the comparison is same inputs, different compiler. That also means the targets clobber each other's output and tsbuildinfo -- this is a benchmark repo, always clean between variant switches:

```shell
rm -rf packages/*/dist
```

## Run

```shell
pnpm install
npx nx run-many -t build           # TS7
npx nx run-many -t build-tsc6      # TS6
```

## Benchmark

```shell
hyperfine --warmup 1 --runs 3 \
  --prepare 'find packages -maxdepth 2 -name dist -type d -exec rm -rf {} +' \
  -n 'build (TS7 tsc)' 'npx nx run-many -t build --skipNxCache' \
  -n 'build-tsc6 (TS6 tsc6)' 'npx nx run-many -t build-tsc6 --skipNxCache'
```

`--skipNxCache` so every run compiles; `--prepare` wipes `dist/` so no run is incremental and neither compiler reads the other's tsbuildinfo.

## Results

Apple Silicon (12 cores, Darwin 25.5.0), Node 24.11.0, nx 23.0.1, TS 7.0.2 vs 6.0.3, pnpm, warm Nx daemon, clean `dist/` before every run, Nx cache skipped. hyperfine, 3 runs each.

### `nx run-many -t build` vs `nx run-many -t build-tsc6`

| Command | Mean [s] | Min [s] | Max [s] | Relative |
|:---|---:|---:|---:|---:|
| `build (TS7 tsc)` | 13.102 ± 0.165 | 12.948 | 13.276 | 1.00 |
| `build-tsc6 (TS6 tsc6)` | 130.676 ± 0.600 | 130.243 | 131.361 | **9.97 ± 0.13** |

TS7 builds the workspace in 13s where TS6 takes 2m 11s. CPU time tells the internal-parallelism story: TS7 spends 53s user across ~4 cores; TS6 spends 164s user on ~1.3 cores (single-threaded checker + GC). The same shape at 25 packages gave 9.95x (33s vs 5m 32s) -- the ratio is per-package, count just scales wall time.
